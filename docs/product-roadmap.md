## Build Philosophy

1. **Each phase ships something usable.** Kein Phase-Ende lässt die App kaputt — nach jeder Phase ist etwas demobar.
2. **Infrastructure before features.** Phase 0 richtet alles ein, damit Feature-Arbeit später schnell und sauber ist.
3. **Magic moment first.** Der Capture-Flow (gesprochener Satz → korrekt einsortiert) funktioniert so früh wie möglich — Ende Phase 1.
4. **Test as you go.** Jeder Task enthält eine Verifikation; Tests nicht ans Ende schieben.
5. **Progressive enhancement.** Erst die simpelste funktionierende Version, dann Politur.
6. **Review before proceeding.** Jede Phase als PR pushen und von einem Review-Agent (z.B. CodeRabbit) prüfen lassen, bevor die nächste startet.
7. **Capture-Reibung ist heilig.** Bei jeder UI-Entscheidung gewinnt der Weg, der das Erfassen verkürzt.
8. **Kein Datenverlust beim Capture.** Rohtext wird immer vor der Klassifikation persistiert.

> Visuelle Tokens sind in `docs/design.md` definiert (YAML-Front-Matter + Prosa). Styling-Werte immer von dort beziehen, nie hart kodieren. Charakter: warm-editorial, ein Rost-Akzent, Serif/Sans/Mono, scharfe Kanten, flach mit Haarlinien.

## Agent Session Guide

### How to Use This Roadmap with Your Coding Agent
1. **Start a session:** Gib dem Agent den Phase-Prompt am Anfang jeder Phase.
2. **Read selectively:** Nur die in der Phase gelisteten Reference sections lesen, nicht die ganzen Dokumente.
3. **Let it work:** Agent findet den ersten ungehakten Task, implementiert, hakt ab.
4. **One session = one phase (ideally):** Möglichst eine Phase pro Session; sonst am letzten ungehakten Task fortsetzen.
5. **Push a PR for review:** Phase fertig → PR pushen, von CodeRabbit prüfen lassen, erst dann nächste Phase.
6. **Need more context?** Referenziert ein Task einen nicht gelisteten Abschnitt, genau diesen on demand lesen.

### Session Tips
- **Don't read everything** — Reference sections sagen exakt, was zu lesen ist.
- **Don't skip tasks** — Reihenfolge löst Abhängigkeiten auf.
- **Verify after each phase** — App nach jeder Phase laufen lassen.
- **Review before moving on** — PR pro Phase, erst nach Merge weiter.
- **Update the status line** — `**Status:** X/58 tasks complete` und `**Current Phase:**` pflegen.
- **`docs/design.md` ist die Styling-Quelle** — alle Farb-, Typo-, Radius- und Spacing-Werte von dort beziehen, nie hart kodieren.
